# NEXT PICK — 프로젝트 컨텍스트

> 이 문서는 Claude Code가 작업 맥락을 이어받기 위한 브리핑 문서입니다.
> 제품의 방향, 현재 상태, 다음 작업, 그리고 반드시 지켜야 할 원칙이 담겨 있습니다.

---

## 1. 제품 개요

**NEXT PICK** — 한국 주식 스크리너 웹앱.

- 한국 주식 약 2,558개 종목을, **성장주 팩터** 방법론을 국내 시장에 응용한 **7개 팩터**로 스코어링한다.
- 각 종목은 종합 **SCORE(0~100)** 와 등급(A+, A 등)을 가진다.
- 핵심 가치 제안: "팩터 점수로 오늘의 주도주(선도주) 후보를 골라준다."
- 배포 상태: Vercel에 배포됨 (https://k-stock-score.vercel.app/). 현재 실사용자는 없고, 수정·개선 중.
- 스택(추정): Next.js. **작업 전 코드베이스를 직접 읽고 실제 스택/구조를 파악할 것.**

### 7개 팩터 (팩터 매핑)
| 팩터 | 팩터 | 의미 |
|------|---------|------|
| 실적 | C | 최근 실적 |
| 성장 | A | 이익 성장 |
| 고가 | N | 신고가 |
| 수급 | S | 수급 |
| 선도 | L | 선도주 여부 |
| 기관 | I | 기관 수급 |
| 시장 | M | 시장 방향 (전체 시장 값, 종목별로 다르지 않음) |

### 특기할 지표: "베이스(BASE)"
- 정의: 최근 1년간 52주 고점 근처(-15% 이내)에 머문 거래일 수. 길수록 가격 기반이 탄탄.
- 오닐의 "베이스 형성(bottoming/base building)" 개념을 정량화한 것. 국내 무료 스크리너엔 드문 지표로, **차별점의 씨앗**이 될 수 있음.

---

## 2. 만든 사람 / 사업 맥락

- 개발자 본인: 풀스택 개발자(주력 Vue 3 + Java/Spring Boot, React/Python도 가능). 경력 약 1년 3개월. 이전 이력은 이탈리안 요리사.
- 이 앱은 하루 1~2시간씩 약 4일 만에 만들어 배포까지 완료함. (개발 실행력은 충분히 입증됨.)
- 최종 목표: 회사에 의존하지 않는 소득원을 만들어, 여행하며 사는 삶.
- 수익화 관심: 구독료 모델. (광고는 트래픽 규모 문제로 부적합하다고 판단, 배제.)

---

## 3. 전략 방향 (합의된 내용)

- **광고 모델은 배제.** 개인 규모로는 유의미한 수익 불가.
- **구독료가 핵심 경로.** 단, 실사용자 0명 상태에서 구독 모델부터 설계하는 건 순서가 틀림.
- **진짜 차별점 / 구독 근거는 "이 SCORE가 실제로 맞다"는 증명에서 나온다.** 즉 백테스트로 예측력을 검증하는 것이 최우선 과제.
- 디자인 개선은 "부끄럽지 않을 정도"의 한 라운드만. 완벽주의로 무한히 다듬지 말 것.
- 검증 후 커뮤니티에 무료 공유 → 사용자 반응으로 차별점을 다듬고 초기 사용자를 모은다 → 그 다음 유료화.

## 4. 지금 진행 중인 작업 (2트랙)

### 트랙 A — UI/UX 개선 (한 라운드만)
현재 화면(다크 모드 스크리너 테이블)은 완성도가 이미 높음. 갈아엎지 말고 아래만 다듬는다.

우선순위 순:
1. **정보 위계 정리 (최우선)**: 테이블 컬럼이 너무 많아 시선 분산됨. SCORE + 7팩터(핵심)를 부가 데이터(종가/등락률/시간외/고점비/거래량/거래대금/외인/기관/프로그램/시총/베이스)와 시각적으로 분리하거나, 부가 컬럼은 접었다 펴게. 목표: "SCORE 높은 종목"을 3초 안에 찾게.
2. **시장(M) 컬럼 중복 제거**: 모든 행에 동일 값(예: 27) 반복 표시 중. 시장 전체 값이므로 테이블에서 빼고 헤더/배너로 이동.
3. **점수 색 스케일 개선**: 팩터 히트맵 녹색이 탁해 고득점/중간점 구분 약함. 고득점이 확 튀게 대비 강화 + 색맹 접근성 고려.
4. **툴팁 일관성**: "베이스" 컬럼엔 이미 툴팁 있음. 7팩터와 주요 부가 컬럼에도 같은 스타일 툴팁 추가.
5. **랜딩/온보딩**: 처음 방문자가 바로 빽빽한 테이블에 던져짐. 상단에 서비스 설명 한 줄 히어로/가이드 추가(기존 사용자에겐 접히게).
6. **모바일 반응형**: 넓은 테이블이 모바일에서 깨짐. 카드형 전환 필요.
7. (여유 시) **개별 종목 상세**: 클릭 시 7팩터를 레이더 차트로, 강점/약점 한눈에.

원칙: 절제된 다크 금융 룩 유지. 과한 애니메이션/그라데이션 금지. 스코어링 로직·데이터 처리는 건드리지 말고 UI 레이어만.
참고: TradingView 스크리너(필터/테이블 UX), Simply Wall St(점수 시각화), Koyfin(타이포/레이아웃)의 강점만 차용.

### 트랙 B — 백테스트 (진짜 핵심, 최우선 검증 과제)
가설: "SCORE가 높은 종목은 이후 기간 동안 시장보다 더 오른다." 이걸 데이터로 검증한다.

**반드시 1단계(데이터 진단)부터 하고 사용자 확인을 받을 것.**

#### 1단계: 데이터 가용성 진단 (가장 먼저)
- 과거 SCORE가 날짜별로 저장돼 있는가? 없으면(가능성 높음) 과거 재무·주가 데이터로 과거 시점 SCORE를 재계산해야 함. 가능한지, 어떤 데이터 소스가 필요한지 진단.
- 주가 데이터가 수정주가(액면분할·배당 반영)인가?
- 상장폐지 종목 데이터가 있는가? (없으면 생존 편향 경고)
- **이 앱의 데이터가 어디서 오는지(증권사 API? 크롤링? 유료 데이터?)를 파악해서 보고할 것.** 백테스트 실현 가능성이 여기 달려 있음.

#### 2단계: 방법론 — 함정 방어 필수
- **미래 정보 누출 방지(look-ahead bias)**: 시점 t의 SCORE는 t 이전 공개 데이터만 사용. 재무는 "실제 공시일" 기준 반영.
- **생존 편향 방지(survivorship bias)**: 상장폐지 종목도 포함. 없으면 한계 명시.
- **거래 현실 반영**: 진입가는 시그널 다음 거래일 시가 등. 슬리피지·수수료(편도 0.2%) 옵션.

#### 3단계: 백테스트 로직
- 리밸런싱: 주간/월간 옵션 (기본 주간).
- 점수 구간(bucket)별 분석: 80+, 70~79, 60~69, 60미만. "점수 높을수록 수익률 높음"이 성립하는지가 핵심.
- 미래 수익률: 진입 후 1주/1개월/3개월 각각.
- 벤치마크 대비: KOSPI/KOSDAQ 대비 초과수익(alpha). 절대수익만 보지 말 것.
- 개별 팩터 검증: 7개 팩터 각각의 예측력도 따로 분석. 유효한 팩터 vs 노이즈 구분.

#### 4단계: 결과 지표
점수 구간별로 — 평균 미래 수익률(1주/1개월/3개월), 승률(%), 벤치마크 초과수익, **표본 수(반드시 표기, 적으면 신뢰도 낮음)**, (포트폴리오 관점) 누적수익 곡선·최대낙폭(MDD).

#### 5단계: 정직한 리포트
- 표로 정리 + "SCORE가 유효한가"에 대한 정직한 결론.
- 통계적 유의성, 표본 부족·우연 가능성을 솔직히 평가. **결과를 좋게 포장하지 말 것.**
- 어떤 구간·팩터가 실제로 효과 있는지, 개선 여지 제안.
- 코드는 재현 가능하게(스크립트 하나로 재실행 가능).

---

## 5. 다음 우선순위

1. **트랙 B 1단계(데이터 진단)** 를 가장 먼저. 백테스트가 현실적으로 가능한지부터 판단. → 결과를 사용자에게 보고하고 확인받기.
2. 데이터가 되면 백테스트 진행. SCORE 예측력 검증이 이 제품의 사활을 가름.
3. 트랙 A(UI 개선)는 병행 가능하되 한 라운드만.
4. 검증 완료 후 → 커뮤니티 공유 → 반응 수집 → 유료화 설계.

---

## 6. 작업 시 지켜야 할 원칙

- 스코어링 로직·데이터 처리 코어는 함부로 바꾸지 말 것. (변경 필요 시 먼저 알릴 것.)
- 큰 작업은 한 번에 다 하지 말고, 단계별로 제안하고 사용자 확인을 받으며 진행.
- 백테스트 방법론의 함정(look-ahead / survivorship / 표본 부족)을 항상 경계하고, 결과를 정직하게 보고.
- 완벽주의로 무한정 다듬기보다, "검증 → 공유 → 반응"의 실행 사이클을 우선.
